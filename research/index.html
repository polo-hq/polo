You have no idea what's in your agent's context window right now.

Here's what we found across 8,257 real agent sessions — and what Budge proposes to do about it.

The window is mostly garbage you've already seen

Problem 99.8% of what your agent reads each turn is content it already read. Context grows 6x from session start to end. Nobody measures this.
Proposal window.resolve() gives you visibility into exactly what's in the window at every turn — by source, by type, by age. Before you can fix it you have to be able to see it.
Roadmap
  • Stable trace schema — session ID, turn ID, source kind, resolver family, estimated tokens, cache read/write/input, content length
  • Session viewer — source breakdown per turn, top cost drivers, cache read vs fresh input, what changed this turn
  • Metadata exporters to Budge Cloud
Validate Do developers actually care about this visibility? Is "I can explain why this turn was expensive" valuable enough to instrument?

Stale content accumulates and nobody evicts it

Problem A file read from turn 18 is still in context at turn 103 even though the file was rewritten at turn 36. Nobody tracks this. An estimated 40–50% of context in coding sessions is deterministically evictable — no model inference required.
Proposal Source-typed assembly means window.resolve() knows about the write at turn 36 and doesn't re-include the stale read. Write-invalidation is mechanical bookkeeping, not summarization.
Roadmap
  • Automatic source classifiers — giant payload, duplicate payload, replayed payload, deterministic stale read, superseded source
  • Deterministic rules — read then write same file, directory listing then file creation, repeated identical fetch/query, exact duplicate tool result across turns
  • Cache-aware shadow policy — labels each source keep / candidate_drop_duplicate / candidate_drop_stale, estimates net savings vs cache impact before any enforcement
  • Safe autopilot v1 — enforces only deterministic low-risk actions, full audit log, one-click rollback, kill switch
Validate Is the 40–50% deterministic eviction estimate real at scale? Does removing stale reads improve outcomes or just reduce cost?

When context gets too big, the fix makes it worse

Problem Compaction fires at turn 400 on a window of 112,000 tokens. The summary is 87.9% as large as what it replaced. Context is back to baseline in 96 turns. One real session compacted 27 times. The tools to fix this exist — no agent ever used them.
Proposal If stale content is evicted continuously as sources go invalid, the pressure never builds to the point where compaction needs to fire. The window stays clean without a crisis intervention.
Roadmap
  • Turn-level counterfactuals — baseline vs drop duplicates vs drop stale reads vs preserve prefix / trim tail
  • Project mode flags — observe, shadow, autopilot
  • Private beta — session view, waste drivers, cache-aware recommendations, autopilot toggle, action log
Validate Does continuous eviction actually prevent the compaction trigger from firing? Does it change the growth curve meaningfully? Does it hold across non-coding agent workloads?
